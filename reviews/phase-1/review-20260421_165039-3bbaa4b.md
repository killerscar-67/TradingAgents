## Findings

### HIGH — `intraday_cache_dir` config key is dead code

**File:** `tradingagents/default_config.py:43-45`, `tradingagents/dataflows/interface.py:176-210`

**Issue:** `intraday_cache_dir` is declared in `DEFAULT_CONFIG` with an env override (`TRADINGAGENTS_INTRADAY_CACHE_DIR`), but `grep` confirms it is consumed nowhere in the codebase — not in `interface.get_intraday_bars` and not in `intraday.get_intraday_bars`. When a caller invokes the interface with `cache_dir=None`, the underlying module falls through to its own hardcoded default (`~/.tradingagents/cache/intraday`). The env override is silently ignored.

**Impact:** Any deployment that sets `TRADINGAGENTS_INTRADAY_CACHE_DIR` to redirect cache storage (e.g., a shared NFS mount, a CI temp dir, a Docker volume) gets no effect. Cache determinism appears to hold from the module's perspective but is non-deterministic from the operator's perspective since two processes using different env configs can write to the same physical path.

**Minimal fix** — `interface.py`, inside `get_intraday_bars`:
```python
resolved_cache_dir = cache_dir or config.get("intraday_cache_dir")
return impl(
    ...
    cache_dir=resolved_cache_dir,
    refresh_cache=refresh_cache,
)
```

Similarly wire `intraday_refresh_cache` from config when the parameter is its default `False` — or document that the config key is intentionally unused at the interface layer and remove it from `DEFAULT_CONFIG` to avoid confusion.

### MEDIUM — No TTL on intraday parquet cache; stale partial data returned silently

**File:** `tradingagents/dataflows/intraday.py:80-98`, `285-345`

**Issue:** The intraday cache has no expiry mechanism. If `end` is a future or same-day date, the first fetch saves a partial DataFrame. All subsequent calls with identical `(symbol, interval, start, end, session)` return the cached partial data indefinitely unless the caller explicitly sets `refresh_cache=True`. No error, warning, or metadata is surfaced to indicate the cached data may be incomplete.

**Impact:** Phase 2's quant engine will call `get_intraday_bars` with today's date as `end`. If the cache was populated mid-session, all subsequent intraday analyses for that date see a frozen, shorter bar series. This is a "quiet stale data" class of bug — the strategy clock advances but the data does not.

**Minimal fix:** When `end` equals or exceeds today's date, skip the cache read (or log a warning). A one-liner guard in `get_intraday_bars`:
```python
today = datetime.now(timezone.utc).strftime("%Y-%m-%d")
if not refresh_cache and end >= today:
    df = None  # never trust cache for live/current date ranges
else:
    df = _load_cache(path) if not refresh_cache else None
```
The handoff acknowledges this as a known limitation; the concern is that Phase 2 callers have no indication they need `refresh_cache=True` for live-date queries.

### LOW — Cache key truncation inconsistency

**File:** `tradingagents/graph/prefilter.py:32`, `tradingagents/dataflows/intraday.py:71`

**Issue:** `prefilter.py` truncates SHA-256 to 16 hex chars (64 bits); `intraday.py` uses 20 hex chars (80 bits). Both are collision-safe for realistic workloads, but the inconsistency creates maintenance confusion.

**Impact:** None in practice. Birthday collision at 64 bits requires ~4.3B entries to reach 50% probability. No correctness risk.

**Minimal fix:** Align both to 20 hex chars, or extract a shared `_hex_digest(src, length=20)` utility.

## Timezone/Session Correctness Audit

`_align_session` (`intraday.py:114-140`) is correct:

- Open boundary: `(hour > 9) | (hour == 9 & minute >= 30)` — 09:30 ET inclusive ✓
- Close boundary: `hour < 16` — 16:00 ET exclusive ✓
- DST: `pytz.timezone("America/New_York")` + `tz_convert` handles spring-forward and fall-back correctly. NYSE regular hours (09:30–16:00 ET) do not span the 1:00–2:00 AM DST hour so no bar duplication or gap at the transition ✓
- Weekday filter: `dayofweek < 5` correctly excludes Saturday/Sunday ✓
- DST spring-forward test (`test_regular_session_around_dst_springforward`) validates the 2026-03-09 Monday boundary ✓

No DST fall-back test exists, but regular-hours filtering is not affected by the transition hour so this is acceptable.
