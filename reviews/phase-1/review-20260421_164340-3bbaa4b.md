#### HIGH — Partial 4h candles pass through silently

**File:** `tradingagents/dataflows/intraday.py:158–177` (`_resample_1h_to_4h`)

**Issue:**
`df.resample("4h", closed="left", label="left")` anchors to UTC midnight. NYSE regular-session 1h bars begin at 10:00 ET, not on a midnight-aligned 4h boundary. In EDT (UTC-4), bars 14:00 and 15:00 UTC land in the `[12:00, 16:00)` window → first 4h candle has 2 constituent bars out of 4. In EST (UTC-5), bars land such that the first window has **1 bar** and the last has **1 bar**, with only the middle window being full. `dropna(how="all")` does not filter these — it only drops all-NaN rows, which no constituent partial candle ever is.

Phase 2 quant engine receives 4h bars with inconsistent completeness and no metadata to distinguish a 1-bar candle from a full 4-bar candle.

**Impact:**
OHLCV values for the first (and last in winter) 4h bar of every regular-session day are computed from fewer than 4 hours of market data. A regime classifier or HTF bias filter consuming these bars may generate a structurally incorrect signal for every morning session.

**Minimal fix:**
After resampling, compute constituent bar counts and drop candles below a threshold. This is a 3-line addition:

```python
resampled = df.resample("4h", closed="left", label="left").agg(**ohlcv)
counts = df["Close"].resample("4h", closed="left", label="left").count()
resampled = resampled[counts >= 2]          # drop partial open/close candles
return resampled.dropna(how="all")
```

A threshold of `2` is conservative (keeps 2-bar candles); `4` is strict (full candles only). Either choice must be documented as a contract for Phase 2. Add a test that counts 4h bars per day for a known date range and asserts the expected count.

#### MEDIUM — `intraday_default_session` config key is dead (never read)

**File:** `tradingagents/dataflows/interface.py:183`, `tradingagents/default_config.py:47`

**Issue:**
`DEFAULT_CONFIG` exposes `intraday_default_session` (env override `TRADINGAGENTS_INTRADAY_SESSION`). However, `interface.get_intraday_bars` hardcodes `session: str = "regular"` and never reads the config key. Setting `config["intraday_default_session"] = "extended"` has no observable effect.

**Impact:**
An operator following the config documentation who sets `TRADINGAGENTS_INTRADAY_SESSION=extended` gets silent no-op. Crypto callers expecting the config default to take effect must pass `session="crypto"` explicitly or the data will be wrong.

**Minimal fix** (choose one):
- Wire it: in `interface.get_intraday_bars`, replace `session: str = "regular"` default with `session: str = None`, then resolve `session = session or config.get("intraday_default_session", "regular")`.
- Or remove the config key entirely and document that all callers must pass `session` explicitly (update handoff accordingly).

#### LOW — No DST transition coverage in session alignment tests

**File:** `tests/test_intraday.py:99–125`

**Issue:**
`SessionAlignmentTests` uses only 2026-04-01 (EDT, UTC-4). The EST/EDT clock changes are 2026-03-08 (spring forward) and 2026-11-01 (fall back). On the spring-forward day, the 2 AM hour is skipped; on fall-back, it repeats. Neither transition is covered.

**Impact:**
The `_align_session` implementation uses `pytz.timezone("America/New_York")` with `tz_convert` which correctly handles DST, so no bug is expected. The risk is a regression if the conversion logic is ever touched. Missing coverage would not catch it.

**Minimal fix:**
Add one test with a timestamp near the March 2026 DST boundary to confirm no bar duplication or omission at the session open:

```python
def test_regular_session_around_dst_springforward(self):
    # 2026-03-08: clocks jump 2 AM → 3 AM ET (spring forward)
    # 09:30 ET = 13:30 UTC on this day (still EST → EDT transition at 2 AM)
    df = _sample_df(["2026-03-08 14:30",  # 09:30 ET
                     "2026-03-08 16:00",  # 11:00 ET
                     "2026-03-08 21:00"]) # 16:00 ET — excluded
    result = _align_session(df, "regular")
    self.assertEqual(len(result), 2)
```
