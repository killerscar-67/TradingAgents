#### HIGH — Live-date cache pollution breaks cross-day determinism
**File**: `tradingagents/dataflows/intraday.py:339–355`

```python
use_cache = (not refresh_cache) and (not _is_live_end_date(end))
if use_cache:
    df = _load_cache(path)

if df is None:
    df = fetch_intraday_bars(...)
    _save_cache(path, df)          # ← called unconditionally when df was None
```

**Issue**: When `_is_live_end_date(end)` is `True` (today or future end date), `use_cache` is `False` and the cache is bypassed for reading — but `_save_cache` is still called because `df is None`. A partial intraday payload (e.g., bars only through 11:00 ET fetched at mid-day) is written to the permanent parquet cache. On any subsequent call after `end` becomes a historical date (i.e., the next calendar day), `_is_live_end_date` returns `False`, the stale partial file is loaded, and the truncated DataFrame is returned as authoritative history.

**Impact**: Violates the Phase 1 acceptance criterion "Cache returns identical DataFrame on repeat call with same key." A caller using `end=today+1` for live bars will silently receive incomplete historical data on the following day. No test covers this cross-day path.

**Minimal fix** (one-line guard):
```python
if df is None:
    df = fetch_intraday_bars(...)
    if not _is_live_end_date(end):   # only persist closed historical ranges
        _save_cache(path, df)
```

#### MEDIUM — `_align_session` regular-session filter uses hour-only comparison; minute-precision bars at :30 boundary depend on a disjunction
**File**: `tradingagents/dataflows/intraday.py:141–143`

```python
hour_mask = (idx_et.hour > 9) | ((idx_et.hour == 9) & (idx_et.minute >= 30))
hour_mask &= idx_et.hour < 16
```

**Issue**: The expression is correct for 15m bars today, but the two-clause disjunction is fragile: any future extension to sub-minute intervals (or a vendor returning second-precision timestamps) would silently pass 09:00:30–09:29:59 ET as "regular session" because `hour == 9` and `minute >= 0` satisfies neither `hour > 9` nor `minute >= 30`. Additionally the upper bound `idx_et.hour < 16` is correct for 15m (last bar starts at 15:45), but a bar at exactly 16:00:00 ET passes this check only if its `hour` field happens to equal 15 — which is fine for hourly resampled bars but worth asserting explicitly.

The more robust form also makes the intent clear:
```python
time_mask = idx_et.time() >= datetime.strptime("09:30", "%H:%M").time()
time_mask &= idx_et.time() < datetime.strptime("16:00", "%H:%M").time()
```
`pd.DatetimeIndex.time` returns `datetime.time` objects, supporting proper sub-minute comparison.

**Impact**: Correctness risk for future interval additions; the present 15m/4h scope is unaffected, but the guard should be tightened before Phase 2 adds more granular signals.

**Minimal fix**: Replace the hour/minute disjunction with a `.time()` range comparison as shown. Requires `from datetime import time` (already imported as `datetime`).

#### MEDIUM — 4h UTC-bucket misalignment yields 0–1 usable candles per NYSE regular-session day
**File**: `tradingagents/dataflows/intraday.py:191`

```python
resampled = df.resample("4h", closed="left", label="left").agg(**ohlcv)
```

**Issue**: `resample("4h")` anchors to midnight UTC (buckets: 00:00, 04:00, 08:00, 12:00, 16:00, 20:00 UTC). For NYSE regular session during EST (UTC-5), the trading day spans 14:30–21:00 UTC. The count filter `>= 2` discards the partial leading bucket (14:30 → 1 bar) and the partial trailing bucket (20:00 → 1 bar), leaving at most one full 4h candle per day (16:00–20:00 UTC). During EDT (UTC-4), the window shifts to 13:30–20:00 UTC and the result is the same or zero candles. The existing test (`test_resample_1h_to_4h_drops_single_bar_partial_windows`) confirms this filtering but passes because the test itself only checks that partial boundary candles are discarded — it does not validate that downstream callers receive useful 4h bars for stocks.

**Impact**: Phase 2 quant engine should not rely on `interval="4h"` with `session="regular"` for equity signals; it will routinely receive 0 or 1 bars per day. This is a design constraint, not a runtime error, but it must be documented before Phase 2 authors take a dependency.

**Minimal fix**: Add a note in `get_intraday_bars`' docstring — and in `_resample_1h_to_4h` — that UTC-aligned 4h buckets are best suited for 24h crypto sessions; for regular-session equities, callers should use `session="extended"` or `session="crypto"`.
