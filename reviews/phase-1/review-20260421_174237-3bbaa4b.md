#### HIGH

**H-1 — `vectorbt` listed as required dependency but gracefully absent in code — prefilter fails silently**

`pyproject.toml` declares `vectorbt>=0.28.1` as a hard dependency, yet `quant_tools.py` wraps the import in `try/except ImportError` and returns `{"error": "vectorbt is not installed"}` if it is missing. `vectorbt` has notoriously complex build requirements on some platforms (NumPy ABI pinning, Numba) and can silently fail to install in constrained envs.

- Files: `tradingagents/agents/utils/quant_tools.py:9-11`, `pyproject.toml`
- Impact: When vectorbt is absent, every `get_quant_signals` call returns an error payload. `score_tickers_with_quant` pushes all tickers to the error bucket; `selected` is empty. The CLI prints "No tickers qualified in quant prefilter" with no explanation. The operator has no indication that a dependency is broken.
- Fix (choice required): Either (a) remove the `try/except` so `ImportError` surfaces at import time (consistent with declaring it required), or (b) move vectorbt to optional deps in `pyproject.toml` and add a startup log warning when the import fails.

#### MEDIUM

**M-1 — `interface.get_intraday_bars` drops the `vendor` parameter**

The public interface wrapper (`tradingagents/dataflows/interface.py:176`) has no `vendor` argument. The underlying `intraday.get_intraday_bars` exposes `vendor: str = "yfinance"`. Phase 2 callers importing from `interface` cannot override the vendor per-call — they must mutate the global config. This deviates from the contract stated in the Phase 1 handoff.

- File: `tradingagents/dataflows/interface.py:176-212`
- Impact: Low in practice (only yfinance is implemented), but the interface is narrower than the contract, which may force Phase 2 to import directly from `intraday` instead of going through the standard routing layer.
- Fix: Add `vendor: Optional[str] = None` to the signature. When non-`None`, pass it directly; otherwise fall through to config routing.

---

**M-2 — 4h UTC-midnight bucket alignment yields ≤1 NYSE regular-session candle per day**

`_resample_1h_to_4h` uses `resample("4h")` anchored at UTC midnight. For NYSE regular session (15:00–21:00 UTC in EST), this produces three buckets: 12:00–16:00 (1 bar), 16:00–20:00 (4 bars), 20:00–00:00 (1 bar). The `_MIN_BARS_PER_4H_CANDLE=2` filter drops the two single-bar boundary buckets, leaving exactly one 4h candle per NYSE day. A 4h interval over a 10-day window returns 10 rows.

- File: `tradingagents/dataflows/intraday.py:168-195`
- Impact: Phase 2 quant callers using `interval="4h"` for NYSE stocks will receive near-daily granularity, not 4h granularity. Any regime classifier or HTF-bias check that expects multiple bars per day will behave unexpectedly.
- Fix: Document clearly in `get_intraday_bars` docstring that `interval="4h"` is only meaningful for `session="crypto"`. Alternatively, use ET-anchored resampling (`offset="9h30min"`) for regular session; this requires passing the session timezone into the resampler.

#### LOW

**L-1 — `_save_cache` silently swallows all exceptions**

```python
def _save_cache(path: Path, df: pd.DataFrame) -> None:
    try:
        df.to_parquet(path)
    except Exception:
        pass  # cache write failure is non-fatal
```

If pyarrow is missing at runtime or the cache directory is read-only, every call silently re-fetches. The `pass` leaves no diagnostic trail.

- File: `tradingagents/dataflows/intraday.py:94-98`
- Impact: Not a correctness issue (data is still correct), but silent failure mode makes performance regressions from repeated re-fetches impossible to diagnose.
- Fix: `import logging; logging.getLogger(__name__).warning("intraday cache write failed: %s", e)` inside the except.

---

**L-2 — 09:30 ET open bar excluded from 1h regular-session filter — not documented in function signature**

The `_align_session` regular-session filter uses `(hour > 9) | (hour == 9 & minute >= 30)`. yfinance 1h bars are labeled at the top of the hour; the bar covering 09:00–10:00 ET has `minute=0` and is excluded. The first included 1h bar is 10:00 ET. This is documented in the handoff known-limitations section but not in the docstring of `_align_session` or `get_intraday_bars`.

- File: `tradingagents/dataflows/intraday.py:123-149`
- Impact: Phase 2 callers passing `interval="4h"` and expecting the 09:30 open in the first 4h candle will be surprised. No code correctness issue.
- Fix: Add one sentence to the `_align_session` docstring: "For 1h yfinance data the bar labeled 09:00 ET (covering the 09:30 open) is excluded; the first regular-session bar is 10:00 ET."

### Code logic verdict (excluding delivery issues)

The core logic in `intraday.py` is correct:

- **Timezone/DST**: `pytz.timezone("America/New_York")` with `tz_convert` handles DST shifts correctly. Spring-forward and fall-back tests are present and exercise the right UTC offsets.
- **No-lookahead**: `df[df.index < cutoff]` (strict less-than) correctly strips bars at exactly `as_of`. Test `test_bars_at_or_after_as_of_are_stripped` confirms this.
- **Cache determinism**: SHA-256 truncated to 20 hex chars (80 bits) is collision-safe for realistic ticker/date/interval cardinality (birthday probability < 10⁻¹⁶ for 10k entries).
- **Live-date guard**: `is_live` captured once before the read-or-fetch-or-write sequence; TOCTOU is resolved.
- **Session close boundary**: `hour < 16` correctly excludes the 16:00 ET bar.
- **4h partial candle filter**: `counts >= _MIN_BARS_PER_4H_CANDLE` correctly drops single-bar boundary windows.
