  MEDIUM — M1: BacktestRequest.config dict leaks through as request.config.execution_mode = 
  "llm_assisted"                                                                                      
   
  File: tradingagents/web/routes/workflow.py:211-216                                                  
  Issue: create_backtest forces payload["execution_mode"] = "quant_strict" at the top level of the
  response request dict, but the caller's config sub-dict (e.g., {"execution_mode": "llm_assisted"})  
  is not cleared. The final request shape returned is:          
  {                                                                                                   
    "config": {"execution_mode": "llm_assisted"},                                                     
    "execution_mode": "quant_strict",            
    "llm_constructed": false                                                                          
  }                                                             
  The top-level execution_mode wins in the current skeleton, but Phase 11 implementers reading
  payload["config"]["execution_mode"] will get the user-supplied override, silently breaking the
  safety invariant.                                                                                   
  Impact: High risk of Phase 11 accidentally running LLM-assisted backtests if it reads from config
  rather than the top-level field.                                                                    
  Minimal fix: Strip execution_mode from the forwarded config in the response so the contract is      
  unambiguous:
  def create_backtest(req: BacktestRequest) -> Dict[str, Any]:                                        
      payload = req.model_dump()                                
      payload.pop("config", None)          # don't leak caller's config dict                          
      payload["execution_mode"] = "quant_strict"                            
      payload["llm_constructed"] = False                                                              
      return _planned("backtest_run", payload, backtest_id="phase-9-contract")
  Or at minimum strip the hazardous key: payload.get("config", {}).pop("execution_mode", None).

  MEDIUM — M2: stream_market_live calls websocket.close() in finally after WebSocketDisconnect        
                                                                                                      
  File: tradingagents/web/routes/workflow.py:156-172            
  Issue: When the client disconnects, the except WebSocketDisconnect: return path still executes the  
  finally: await websocket.close() block before returning. Calling close() on an already-disconnected 
  websocket raises a RuntimeError or WebSocketDisconnect in Starlette, which is suppressed only if    
  nothing catches it at the framework level — but it may generate spurious error logs in production.  
  Impact: No data loss, but error noise and potential unhandled-exception logs for every client
  disconnect.                                                                                         
  Minimal fix:
  async def stream_market_live(websocket: WebSocket) -> None:                                         
      await websocket.accept()                                  
      try:                    
          await websocket.send_json({...})                                                            
          await asyncio.sleep(0.01)       
      except WebSocketDisconnect:                                                                     
          return                                                                                      
      else:     
          await websocket.close()                                                                     
  Or track state with a flag: connected = True; finally: if connected: await websocket.close().
