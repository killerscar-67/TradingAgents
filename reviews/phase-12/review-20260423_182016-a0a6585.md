## Findings

### đź”´ HIGH â€” POST `/api/backtests` response is missing `strategy_id`, `start_date`, `end_date`

**File**: `tradingagents/web/routes/workflow.py` `create_backtest()` (~line 296)  
**File**: `web/src/screens/BacktestScreen.tsx` `BacktestApiResponse` interface (~line 32)

The route returns `{ kind, status, backtest_id, workflow_session_id, result, request }`. The frontend `BacktestApiResponse` declares `strategy_id`, `start_date`, and `end_date` as required strings. After the POST, `run.strategy_id`, `run.start_date`, and `run.end_date` are silently `undefined`.

No crash today because these fields are not rendered in `BacktestScreen`, but the test stub masks the gap â€” it includes `strategy_id` in the mock response while the real backend does not emit it. The type safety guarantee is broken.

**Fix options** (pick one):
1. Add the missing fields to the route response: `"strategy_id": backtest["strategy_id"], "start_date": req.start_date, "end_date": req.end_date`.
2. Mark the fields optional in `BacktestApiResponse` (`strategy_id?: string`, etc.) and update `mapBacktestResponseToRun` to fall back gracefully.

### đź”´ HIGH â€” `GET /api/backtests/{backtest_id}` is dead code in the normal flow

**File**: `web/src/screens/BacktestScreen.tsx` re-fetch `useEffect` (~line 161)

```ts
if (!done || !backtestId || run?.backtest_id === backtestId) return;
```

The POST handler calls `setRun(mapBacktestResponseToRun(data))`, which sets `run.backtest_id = backtestId`. When the SSE stream completes and `done` becomes `true`, the guard short-circuits immediately because `run.backtest_id === backtestId` is already true. The new `GET` endpoint is never called in the happy path.

Currently safe because the backend runs the backtest synchronously and the POST result is final. But the polling intent â€” hydrating a persisted result on revisit or if the POST result is stale â€” is never exercised, and no test covers it.

**Fix options** (pick one):
1. Change the guard: `if (!done || !backtestId || run?.status === "completed") return;` so the GET fires after SSE completes regardless of whether a run is already set.
2. Remove the guard entirely and always re-fetch when `done && backtestId`.
3. Keep the current guard and add a test that simulates arriving at the screen with an existing `backtestId` (no POST result) and verifies the GET is called.

### đźźˇ MEDIUM â€” `BatchScreen` drops all but the last SSE event per render cycle

**File**: `web/src/screens/BatchScreen.tsx` (~line 33)

```ts
const event = events[events.length - 1];
```

The SSE hook appends each arriving event to the `events` array. When React 18 batches several state updates into one render, the `useEffect` processes only the most recently appended event. For a basket of 10+ symbols delivered in a burst, intermediate `batch_item` events are silently skipped and the corresponding result cards stay stale until the next flush.

**Fix**: Track the last processed index with a `useRef`, then slice and process `events.slice(lastIdx.current)` in the effect.
