## Findings by Severity

### MEDIUM

**M-1 — `BacktestTrade.net_pnl` is overstated by `commission` for EOD exits**

For end-of-data closes the equity simulation is correct:
- Entry: `realized_equity -= commission` (entry commission deducted)
- EOD exit: `realized_equity += gross` (no exit order placed)
- Actual equity gain = `gross − commission` ✅

However the trade record stores `net_pnl = round(net, 6)` where `net = gross`. This overstates `net_pnl` by exactly `commission` (default $1.00) because the entry commission already paid is not subtracted:

```python
# actual equity gain: 19.0
# BacktestTrade.net_pnl recorded: 20.0  ← overstated by commission
```

The `commission` field on the trade record correctly says `1 × commission_per_order` was paid, making the record internally inconsistent: `net_pnl ≠ gross_pnl − commission`. The handoff acknowledges "slightly overstates net_pnl for EOD positions" but this is not a small rounding error — it is exactly `commission_per_order`. Fix: set `net_pnl = round(gross - commission, 6)` for EOD exits (matching the actual equity path).

**M-2 — OOS walk-forward folds start cold: no IS 4h context available**

In `run_walk_forward`, the OOS 4h bars are sliced strictly to the OOS window:

```python
oos_4h = bars_4h.loc[
    (bars_4h.index >= bars_15m.index[oos_start])
    & (bars_4h.index <= bars_15m.index[oos_end - 1])
]
```

Within the OOS `run_backtest`, at bar `i=0` the engine sees `visible_4h = bars_4h.loc[bars_4h.index <= oos_15m.index[0]]`. Since `oos_4h` contains only bars from the OOS period, `visible_4h` is empty at the start of each fold and stays below `min_4h_bars` (default 30) for the first ~120 hours of OOS time. The engine emits `NoSignal` for all those bars.

This makes OOS backtests significantly more conservative than IS backtests (which also start cold but over a shorter warmup). OOS Sharpe will be structurally lower at fold boundaries. The issue is not a lookahead violation but a design choice that biases walk-forward metrics. Documented in known limitations; flagged here because it materially affects the validity of `oos_sharpe_positive_pct` as a generalization metric. Recommend passing IS 4h tail as pre-history to the OOS backtest, or documenting this as a known pessimistic bias.

### LOW

**L-1 — `_empty_result` breaks the `len(equity_curve) == len(bars_15m)` contract for `n_bars < 2`**

When `len(bars_15m) < 2`, the function returns early with `equity_curve = ()`. For `n_bars == 1` this means `len(equity_curve) = 0 ≠ 1`. The handoff states "equity curve has exactly `len(bars_15m)` entries" as a contract. The `_empty_result` path violates it. In practice no production caller should pass a single-bar DataFrame, but the invariant should hold universally or the contract should be narrowed to `n_bars ≥ 2`.

**L-2 — Stop fill uses next-bar open plus slippage, but stop trigger uses current bar's Low/High**

Stop logic:
- Trigger: `bar_low <= pos_stop_price` at bar `i` (intra-bar touch)
- Fill: next-bar open with slippage at bar `i+1`

When the stop is hit intra-bar, the actual exit price could be below (long) or above (short) the next bar's open. The model always fills at next-bar open, which may be higher than the stop for a gapping-down long. This is documented as a known simplification (no gap-fill modeling) and is consistent with the Phase 4 paper adapter behavior. Flagged for completeness; no code change required.
