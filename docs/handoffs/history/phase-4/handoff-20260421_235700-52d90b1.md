## Runtime contract

`TradingAgentsGraph.build_order_intent(symbol, trade_date, final_trade_decision_text, quant_contract=None, risk_context=None) -> dict`

When `execution_mode == "quant_strict"`:

- Order rating is still derived from `QuantSignalContract`, not LLM text.
- If `risk_context` is omitted or incomplete, the method returns the existing strict-mode order intent without sizing annotations.
- If `risk_context` is complete, the method calls:
    - `size_position(entry_signal, entry_price, atr_15m, account_equity, config)`
    - `check_risk_gates(size_contract, daily_loss_state, current_exposure, account_equity, config)`
- Returned `order_intent["annotations"]["risk"]` contains:
    - `applied`: `True`
    - `size_contract`: serialized `PositionSizeContract`
    - `gate`: serialized `RiskGateResult`
- If sizing inputs are invalid, the order intent is blocked with `reason="Risk sizing failed: ..."` and `annotations["risk"]["applied"] == False`.

Expected `risk_context` keys:

- `entry_signal`: `EntrySignal` or dict with `engine`, `direction`, `strength`, `reason`
- `entry_price`: positive float
- `atr_15m`: positive float
- `account_equity`: positive float
- `current_exposure`: float, defaults to `0.0`
- `daily_loss_state`: `DailyLossState` or dict with `date`, `net_pnl`, `kill_switch`, `trade_count`; defaults to `DailyLossState.new_day(trade_date)`

## Contracts exposed to next phase

- `OrderIntentContract` remains the single execution handoff envelope. Phase 4 stores risk output under `annotations["risk"]` to avoid changing the Phase 0 contract shape.
- `annotations["risk"]["size_contract"]` is a serialized `PositionSizeContract` with the actual ticker symbol.
- `annotations["risk"]["gate"]` is a serialized `RiskGateResult`. Downstream order submission must drop the order when `allowed == False` or `order_intent["blocked"] == True`.
- `config["risk_context"]` may be used as a graph-level fallback when `risk_context` is not passed directly to `build_order_intent`.

## Known limitations / deferred decisions

- No live broker adapter is implemented. Phase 4 produces a deterministic, risk-gated order intent but does not submit orders.
- No portfolio reconciliation loop is implemented. Callers must supply `current_exposure` and `daily_loss_state`.
- No automatic ATR computation is performed in `build_order_intent`; callers must compute `atr_15m` from 15m bars before passing `risk_context`.
- Incomplete `risk_context` leaves the existing strict-mode intent unchanged rather than blocking. Callers that require hard gating must enforce complete context at the call site.
- `propagate(...)` does not yet accept `risk_context`; direct callers can use `build_order_intent(...)`, or future work can thread `risk_context` through `propagate` and `propagate_prefiltered_universe`.

## What the reviewer must focus on

- Verify `quant_strict` still never uses LLM text for executable rating extraction.
- Verify risk annotations are only attached when enough deterministic inputs are present.
- Verify invalid sizing inputs block the order instead of producing a partially actionable intent.
- Verify exposure and daily-loss state are supplied by the caller and not inferred from stale or mutable graph state.
- Decide whether the next phase should make incomplete `risk_context` a hard block in strict mode.
